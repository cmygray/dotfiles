# GitHub Issue 생성 워크플로우

## 실행 지침

이 커맨드가 호출되면:

1. **대화 컨텍스트 분석**
   - 현재 대화에서 언급된 버그, 기능 요청, 개선사항 등을 식별
   - 이슈 타입 추론 (bug/feature/enhancement/documentation 등)

2. **이슈 템플릿 확인**
   - `.github/ISSUE_TEMPLATE/` 디렉토리 존재 여부 확인
   - 이슈 타입에 맞는 템플릿 찾기:
     - `bug_report.md` / `bug.md`
     - `feature_request.md` / `feature.md`
     - `enhancement.md`
     - 기타 프로젝트 특화 템플릿

3. **템플릿 선택**
   - 적합한 템플릿이 1개: 자동 선택
   - 적합한 템플릿이 여러 개: 사용자에게 선택 요청
   - 템플릿 없음: 기본 구조 사용

4. **이슈 초안 작성**
   - 선택된 템플릿 형식에 맞춰 한글로 작성
   - 대화 컨텍스트의 정보를 템플릿에 채워넣기
   - 재현 방법, 예상 동작, 실제 동작 등 구체화

5. **사용자 확인**
   - 작성된 초안을 보여주고 동의 구하기
   - 수정 요청 시 반영
   - 라벨, 마일스톤 등 메타데이터 추가 여부 확인

6. **이슈 생성**
   - `gh issue create` 명령어 사용
   - 한글 제목과 본문 사용
   - 라벨/담당자/마일스톤 지정 (선택사항)
   - 성공 시 Issue URL 반환

## 기본 이슈 구조 (템플릿 없을 시)

### Bug Report
```markdown
## 문제 설명
<!-- 발생한 문제를 명확하게 설명해주세요 -->

## 재현 방법
1.
2.
3.

## 예상 동작
<!-- 어떻게 동작해야 하는지 설명해주세요 -->

## 실제 동작
<!-- 실제로 어떻게 동작하는지 설명해주세요 -->

## 환경
- OS:
- 버전:
- 기타:

## 추가 정보
<!-- 스크린샷, 로그 등 도움이 될만한 정보 -->
```

### Feature Request
```markdown
## 기능 설명
<!-- 제안하는 기능에 대해 설명해주세요 -->

## 동기 및 배경
<!-- 왜 이 기능이 필요한가요? 어떤 문제를 해결하나요? -->

## 제안하는 해결 방법
<!-- 이 기능을 어떻게 구현하면 좋을지 아이디어가 있다면 -->

## 대안
<!-- 고려한 다른 방법들이 있다면 -->

## 추가 정보
<!-- 참고 자료, 예시 등 -->
```

## 주의사항

- 템플릿이 여러 개일 경우 가장 적합한 것 선택
- 충분한 정보가 없으면 사용자에게 추가 질문
- 민감한 정보(API 키, 비밀번호 등) 포함 주의
- GitHub CLI 인증 필요 (`gh auth status`로 확인)

## Git & GitHub 가이드라인

- gh CLI 사용 (웹 인증 문제)
- 이슈 메시지를 한글로 작성
- `.github/ISSUE_TEMPLATE/` 참고할 것

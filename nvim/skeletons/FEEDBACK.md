좋은 개발자란

- 조금씩 깊게 물어봐서 오버엔지니어인지 확인하는

커밋과 PR

- 소통 능력 검증
- 과제를 내면 과제로 파악 가능
- 형식에 대해

애자일/폭포수

- 매니저 급일때

테스트 자동화/TDD

- 자동화와 TDD를 헷갈리는 사람인가
- 자동화 도구와 테스트를 활용해 피드백을 받으면서 점진적 작업을 할 줄 아는 사람인가
- 테스트 대상의 기준
- 테스트 설계 방식

좋은 설계

- 대부분 솔리드 원칙으로 대답함
- 어떤 철학으로 일하는지 파악하기 위해서지 답은 없음

DIP

- DI를 그저 데코레이터로 이해하는 사람이 가끔 있음

소프트웨어 재사용을 위한 느슨한 연동 방법은?

- 패턴 (어댑터; 번역계층 안티 corruption)

IoC

- 스프링땜에 있는 질문

### 백엔드질문

RESTful?

- 어차피 우리도 표현력 딸려서 RESTful 하지 않음
- 본인이 레스트풀하다고 한다면?
  - 레스트풀이 뭔지 모른다
  - 구현이 단순했다

원격 API 제공 방법?

- 요즘 잘 안물어봄

DTO, VO, Entity, DAO?

- 생각보다 구별 못하는사람 있음

유스케이스 패턴?

- 헥사고날 아키텍처

Repository?

- 헥사고날 아키텍처, 포트

맵퍼와 액티브레코드 패턴

- ORM을 프레임워크로만 써본사람
- 도메인과 DB 레이어 구분을 못하는사람

이벤트 아키텍처와 리액티브 시스템

- 고수 전용 질문임1

잘못된 MSA

- 기능중심적 구분
- 데이터베이스를 나누지 않는 경우 \*\*\*
- 의사결정이 어차피 중앙에서 일어나는 경우
- 인터페이스에 의거한 커뮤니케이션을 못하는 경우

Prepared statement?

- 이건 나도 모름

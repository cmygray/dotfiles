# refine-task — 착수 전 태스크 인수 테스트 정제

Notion 태스크(또는 GitHub Issue)의 서술형 요구사항을 Gherkin 인수 테스트로 정제해
페이지에 채워 넣고, 사용자 피드백으로 다듬는다. 구현 착수 전 또는 구현 직후
검증 기준 확정 단계에서 사용한다.

## 워크플로우

### 1. 태스크 + 상위 맥락 읽기

태스크 한 장만 보면 범위·워딩이 빈약하다. **태스크 → 프로그램 → PRD 체인을 끝까지
따라가** 상위 맥락을 확보한 뒤 인수 테스트를 쓴다.

- **태스크 페이지**: Notion MCP(`notion-fetch`)로 읽는다. GitHub Issue면 `gh issue view`.
  - 파악할 것: 가치/완료 기준, 핵심 흐름(→로 이어진 단계 서술), 데모/결과물 정의.
  - 페이지 properties에서 체인 링크를 추출한다: `프로그램`(→ 프로그램 페이지),
    `상위 작업`/`하위 작업`(→ 같은 슬라이스 트리), `에픽`, `GitHub Issue`.
  - **하위 작업의 진행 상태도 확인**한다 — `취소`된 하위작업은 범위 판단에서 제외하되,
    어떤 구현 단위(컴포넌트·탭 등)를 가리키는지는 워딩 참고용으로 읽어둔다.
- **프로그램 페이지**(태스크의 `프로그램` 링크): `notion-fetch`로 읽는다.
  - `PRD` property에서 PRD 페이지 링크를, `작업` property에서 **형제 슬라이스 목록**을 얻는다.
  - 형제 슬라이스 제목들로 "이 항목은 몇 번 태스크 소관"인지 매핑한다 → 4·5단계의
    **범위 제외 항목**을 "어느 슬라이스 소관"까지 정확히 적는 근거가 된다.
- **PRD 페이지**(프로그램의 `PRD` 링크): `notion-fetch`로 읽는다. 여기가 범위·워딩의 원천.
  - `목표`/`Objective`, `Ideal Flow`(Flow A/B/C…), `요구사항` 표를 읽어 도메인 행동을 잡는다.
  - **`참고사항 > 1차 범위`의 `포함`/`제외` 목록이 스코프 경계의 권위 있는 출처**다.
    제외 목록은 그대로 범위 제외 메모의 후보가 된다.
  - PRD 본문에 적힌 사용자 노출 워딩(탭 이름·상태 문구·정렬·표시 규칙)을 우선 채택하고,
    2단계에서 코드로 교차 검증한다.
- 관련 구현이 이미 있으면(브랜치/PR) 코드를 함께 본다 — 인수 테스트는 구현 디테일이
  아니라 도메인 행동 기준으로 쓰되, 실제 동작과의 갭은 별도로 보고해야 하므로.

> 참고: Notion 태스크 페이지에는 `<ancestor-path>`에 상위 DB·페이지 경로가 함께 오지만,
> 이는 보드/DB 계층일 뿐 PRD가 아니다. PRD는 반드시 **프로그램의 `PRD` property**로 찾는다.

### 2. 워딩 검증 (MUST)

시나리오에 쓸 용어는 **제품이 실제 쓰는 워딩**이어야 한다:

- 내부 enum/API 상태명을 직역하지 않는다 (예: `ReviewInProgress` → "검수 중" ❌).
- 코드베이스에서 사용자 노출 문구를 grep해 확인한다 (UI 컴포넌트의 한국어 문자열,
  토스트/안내 메시지 등). 예: "AI가 분석 중이에요" → "AI 분석" 계열 워딩 사용.
- 상태 조건은 구현 상태값이 아니라 도메인 사건으로 표현한다
  (예: "다음 회차 글이 InProgress다" ❌ → "고쳐쓰기 포함 피드백을 받았고 다음 회차
  글을 아직 제출하지 않았다" ✅).

### 3. Gherkin 작성 규칙 (MUST)

- **키워드는 영문**(`Feature` / `Background` / `Scenario` / `Given` / `When` / `Then` / `And`),
  문장은 한국어.
- **한 시나리오 = 한 행동**: When은 시나리오당 하나가 원칙.
  - When→Then→When→Then 교차 반복 금지 → 시나리오를 분리한다 (열기/닫기, 다음/이전 등).
  - 시나리오 중간에 Given 재등장 금지 → 분리한다.
  - And/Then 절에 행동을 숨기지 않는다 ("다시 누르면 닫힌다" ❌).
  - And는 Then 뒤 추가 검증, Given 뒤 추가 전제로만 사용.
- 공통 전제(피처 플래그, 진입 페이지)는 `Background`로.
- 포함할 시나리오 유형 체크리스트:
  - 정상 흐름 (핵심 단계별 1개씩)
  - 피처 플래그/권한 OFF 시 기존 동작 유지
  - 빈 상태 (대상 데이터 없음)
  - 경계 (첫/마지막, 0개/최대)
  - 표시 제외 조건 (예: 점수·피드백 미노출 같은 명시적 비표시 요구)
- 마지막에 **범위 제외 항목**을 명시한다. 출처는 PRD `1차 범위 > 제외` 목록과
  형제 슬라이스 매핑이며, 가능하면 "어느 슬라이스/하위작업 소관"까지 적는다.

### 4. 페이지에 바로 반영

- 초안을 채팅에만 보여주고 "어떻게 할까요?" 묻지 않는다 — **바로 Notion 페이지 끝에
  `# 인수 테스트` 섹션 + ```gherkin 코드 블록으로 추가**하고 링크를 알린다.
  사용자는 페이지에서 보고 피드백을 준다.
- 피드백 라운드: 지적받은 시나리오만 `update_content`(old_str/new_str)로 정밀 수정.
  같은 형식 문제가 다른 시나리오에도 있으면 일관되게 함께 고친다.

### 5. 구현 갭 보고

워딩/조건을 코드로 검증하다가 도메인 기준과 구현이 어긋난 지점을 발견하면
(예: 도메인은 "피드백 받음"인데 구현은 `InProgress`만 체크), 인수 테스트는
도메인 기준으로 쓰고 **갭을 사용자에게 별도 보고**한다. 임의로 코드를 고치지 않는다.

## 산출물

- Notion 태스크 페이지의 `# 인수 테스트` 섹션 (Gherkin 코드 블록 + 범위 제외 메모)
- 발견된 구현 갭 목록 (있을 경우, 채팅으로 보고)
